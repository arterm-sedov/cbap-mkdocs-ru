Введение

Comindware Platform может принимать HTTP-запросы от внешних систем с помощью адаптера импорта данных в платформу.

Адаптер открывает HTTP-порт, принимает запросы по заданному пути URI, проверяет имя пользователя и пароль, преобразует XML-сообщение во внутреннюю структуру DataImport и передаёт данные в сценарий для создания или обновления записей.

Связь с получением сообщений через Kafka

Адаптер для импорта данных решает ту же прикладную задачу, что и получение сообщений через Apache Kafka: принимает данные из внешней системы, преобразует их в сообщение Comindware Platform и запускает сценарий для создания или обновления записей.

Различается способ доставки данных:

  • при получении сообщений через Apache Kafka внешняя система отправляет сообщение в топик, а Comindware Platform считывает его через подключение к шине сообщений;
  • при использовании адаптера импорта данных внешняя система отправляет HTTP-запрос напрямую на порт адаптера, а адаптер передаёт полученное XML-сообщение в механизм DataImport.

Используйте подключение к Apache Kafka, если обмен должен идти через брокер сообщений. Используйте адаптер импорта данных, если внешняя система может отправлять HTTP-запросы и требуется обработать их через пользовательский адаптер с механизмом DataImport.

Логика работы адаптера

  1. Внешняя система отправляет HTTP-запрос на адрес адаптера.
  2. Адаптер проверяет Basic-аутентификацию.
  3. Адаптер проверяет метод запроса и путь URI.
  4. Тело XML-запроса преобразуется в сообщение типа ImportDataPlatformMessage.
  5. Comindware Platform запускает сценарий с событием «Получение сообщения».
  6. Сценарий создаёт или обновляет записи в выбранном шаблоне.

Предварительные требования

  • Адаптер ImportDataPlatformAdapter установлен, опубликован и доступен в списке пользовательских подключений. См. «Адаптеры. Определения, настройка, удаление».
  • Если адаптер отсутствует в списке пользовательских подключений или требуется доработать маппинг под внешний XML-контракт, запросите исходный код или образец адаптера у представителя Comindware.
  • На сервере Comindware Platform разрешён входящий доступ к порту адаптера.
  • Подготовлен шаблон записи для импортируемых данных.
  • Известна структура XML-запроса внешней системы.
  • У пользователя есть права на настройку подключений, путей передачи данных и сценариев.

Формат входящего XML-запроса

Адаптер обрабатывает XML-запросы со структурой заявки.

Согласуйте структуру XML-запроса с внешней системой и используйте понятные имена элементов, отражающие состав передаваемых данных.

Пример входящего XML-запроса:

<Requests> 
    <Request> 
        <ContactName>Иванов Иван</ContactName> 
        <RequestSummary>Краткое описание заявки</RequestSummary> 
        <ResolutionActions>Необходимые действия</ResolutionActions> 
        <RequestCategory> 
            <Code>SUPPORT</Code> 
        </RequestCategory> 
    </Request> 
</Requests>

Рекомендуемое сопоставление полей:

Поле XML Бизнес-поле сообщения Назначение
ContactName ContactName Контактное лицо или источник заявки
RequestSummary RequestSummary Краткое описание заявки
ResolutionActions ResolutionActions Действия, необходимые для обработки заявки
RequestCategory/Code RequestCategoryCode Код категории заявки

Имена полей

Имена полей в сообщении должны быть понятны администраторам приложения и специалистам, которые сопровождают интеграцию. Если внешняя система передаёт XML с другими именами элементов, настройте их сопоставление с атрибутами сообщения.

Настройка подключения

  1. Откройте страницу «Администрирование» — «Инфраструктура» — «Подключения».
  2. Создайте подключение типа «Пользовательские подключения» — «DataImport Platform Adapter».

  3. Настройте свойства подключения:

    • Системное имя — введите уникальное системное имя подключения, например platform_receive_requests_connection. Не должно начинаться с цифры. Разрешены английские и русские буквы, цифры и символ «_». Рекомендуется использовать английские буквы.
    • Отключить — установите этот флажок, если требуется временно отключить подключение.
    • Описание — введите наглядное описание подключения, например «Получение заявок из внешней системы».
    • Запись в файловые журналы — выберите режим записи сведений о работе подключения:
      • Полные сведения об обработке сообщения;
      • Только ошибки;
      • Отключить.
    • Порт — укажите порт, на котором адаптер будет принимать HTTP-запросы, например 8085.
    • Имя пользователя — укажите имя пользователя для Basic-аутентификации.
    • Пароль — укажите пароль для Basic-аутентификации.

  4. Сохраните подключение.

  5. Нажмите кнопку «Проверить соединение».

Внимание!

Порт адаптера должен быть доступен для внешней системы и не должен совпадать с портом сервера Comindware Platform.

Настройка пути передачи данных

  1. Откройте страницу «Администрирование» — «Пути передачи данных».
  2. Создайте путь передачи данных типа «Пользовательские подключения» — «DataImport Platform Adapter».

  3. На вкладке «Основные свойства» настройте параметры:

    • Подключение — выберите подключение адаптера импорта данных.
    • Системное имя — введите уникальное системное имя пути передачи данных, например platform_receive_requests_route. Не должно начинаться с цифры. Разрешены английские и русские буквы, цифры и символ «_». Рекомендуется использовать английские буквы.
    • Отключить — установите этот флажок, если требуется временно отключить путь передачи данных.
    • Описание — введите наглядное описание пути передачи данных, например «Получение заявок по HTTP».
    • Номер шины данных — выберите номер от 0 до 3, если требуется распределить потоки данных нескольких путей для повышения производительности.

  4. На вкладке «Атрибуты сообщений» настройте сопоставление полей сообщения с атрибутами шаблона записи.

    Примечание

    Если системные имена полей запроса совпадают с системными именами атрибутов шаблона, Comindware Platform сопоставит их автоматически. Для явного сопоставления используйте альтернативные названия атрибутов.

  5. На вкладке «Интеграция» настройте параметры HTTP-запроса:

    • Шаблон пути запроса — укажите путь URI без начального слеша, например requests.
    • Метод запроса — укажите HTTP-метод, например POST.

  6. Сохраните путь передачи данных.

Итоговый адрес для внешней системы:

http://<serverHost>:<port>/<requestPath>

Например:

http://<your-host>:8085/requests

Настройка сценария

Для обработки полученных данных настройте сценарий со стартовым событием «Получение сообщения».

  1. Откройте страницу администрирования приложения.
  2. Выберите пункт «Сценарии».
  3. Создайте сценарий, например «Импорт заявок».

  4. Настройте свойства сценария:

    • Контекст выполнения — выберите подходящий контекст выполнения, например «От инициатора».
    • Статус — на время настройки выберите «Приостановлен», после проверки сценария выберите «Активен».

  5. В конструкторе сценария настройте стартовое событие:

    • Тип — «Получение сообщения».
    • Контекстный шаблон — выберите шаблон записи, в который требуется импортировать данные.
    • Подключение — выберите подключение адаптера импорта данных.
    • Путь передачи данных — выберите путь передачи данных адаптера импорта данных.
    • Имя переменной — укажите переменную для полученного сообщения, например importDataMessage.

  6. Добавьте действия для создания или обновления записей на основе полученного сообщения.

  7. Сохраните и активируйте сценарий.

Проверка работы адаптера

Отправьте тестовый HTTP-запрос с Basic-аутентификацией:

curl -X POST "http://<your-host>:8085/requests" \ 
    -u "test_user:test_password" \ 
    -H "Content-Type: application/xml" \ 
    --data-binary "@request.xml"

При успешном получении запроса адаптер вернёт ответ 202 Accepted.

Ограничения

  • Адаптер начинает принимать и обрабатывать HTTP-запросы только после создания и включения подключения и пути передачи данных.
  • HTTP-запросы, отправленные до запуска адаптера или во время отключения подключения либо пути передачи данных, не сохраняются и не обрабатываются после включения.
  • Передача документов в описанном сценарии не поддерживается.
  • Для интеграции с внешней системой и сопоставления нестандартной структуры XML со структурой шаблонов можно разработать пользовательский адаптер. На уровне адаптера можно разобрать входящий запрос, выполнить маппинг и использовать механизм DataImport.

Для использования механизма DataImport в пользовательском адаптере добавьте в файл Adapter.cs следующий метод:

public override PerformerType GetPerformerType() 
{ 
    return PerformerType.DataImport; 
}

Диагностика

Симптом Возможная причина Решение
401 Unauthorized Неверные имя пользователя или пароль Проверьте учётные данные в подключении и HTTP-запросе
405 Method Not Allowed Метод запроса не совпадает с настройкой пути передачи данных Проверьте поле «Метод запроса»
404 Route not found Путь запроса не совпадает с настройкой пути передачи данных Проверьте поле «Шаблон пути запроса»
400 Bad Request Пустое тело запроса или ошибка XML Проверьте XML-запрос и журнал адаптера

Для подробной диагностики просмотрите журналы адаптера и события получения сообщений. См. «Журналы событий. Типы, просмотр, цепочки событий».
