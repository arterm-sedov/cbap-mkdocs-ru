# Регламент ввода в эксплуатацию приложений на основе {{ productName }} {: #commissioning_recommendations }

## Введение {: #commissioning_recommendations_intro }

Здесь представлен рекомендованный порядок ввода в эксплуатацию приложений, разработанных на основе **{{ productName }}**. Регламент описывает этапы от разработки до промышленной эксплуатации и требования к каждому этапу.

См. также _[системные требования {{ productName }}][system_requirements]_ в минимально возможной конфигурации для запуска и опытно-промышленной эксплуатации (ОПЭ) одного приложения на основе **{{ productName }}**.

## Этапы жизненного цикла приложений {: #commissioning_recommendations_stages }

Жизненный цикл приложений на основе **{{ productName }}** включает следующие этапы:

1. Разработка
2. Тестирование
3. Опытно-промышленная эксплуатация (ОПЭ)
4. Нагрузочное тестирование
5. Промышленная эксплуатация
6. Ежегодный пересмотр конфигурации

## Этап 1: Разработка {: #commissioning_recommendations_development }

### Требования к инфраструктуре

- Должна быть предусмотрена отдельная инфраструктура с платформой для разработки.
- Конфигурация должна соответствовать минимальным требованиям, указанным в _«[Системные требования {{ productName }}][system_requirements]»_.
- Рекомендуется использовать конфигурацию с запасным локомотивом (стабильная версия и разработка) для обеспечения возможности параллельной работы стабильной версии платформы и версии для разработки новых функций.

См. _«[Развёртывание ПО. Архитектура и ИТ-ландшафт][architecture_landscape]»_ для получения информации о рекомендуемых конфигурациях серверов разработки и тестирования.

### Критерии готовности к переходу на следующий этап

- Приложение разработано и проходит базовое функциональное тестирование.
- Документация по приложению подготовлена.

## Этап 2: Тестирование {: #commissioning_recommendations_testing }

### Требования к инфраструктуре

- Должна быть предусмотрена отдельная инфраструктура с платформой для тестирования.
- Конфигурация должна соответствовать минимальным требованиям, указанным в _«[Системные требования {{ productName }}][system_requirements]»_.
- Инфраструктура тестирования должна быть изолирована от инфраструктуры разработки и промышленной эксплуатации.

### Объем тестирования

На этапе тестирования должны быть выполнены следующие виды тестирования:

- Функциональное тестирование — проверка соответствия функциональности приложения требованиям.
- Интеграционное тестирование — проверка взаимодействия приложения с внешними системами.
- Регрессионное тестирование — проверка отсутствия регрессий после изменений.
- Тестирование безопасности — проверка соответствия требованиям безопасности.

### Критерии готовности к переходу на следующий этап

- Все виды тестирования пройдены успешно.
- Критические ошибки устранены.
- Приложение готово к опытно-промышленной эксплуатации.

## Этап 3: Опытно-промышленная эксплуатация (ОПЭ) {: #commissioning_recommendations_ope }

### Требования к инфраструктуре

- Конфигурация должна соответствовать минимальным требованиям, указанным в разделе _«[Требования к техническому обеспечению][system_requirements_hardware]»_ статьи _«[Системные требования {{ productName }}][system_requirements]»_.
- Инфраструктура ОПЭ должна быть изолирована от инфраструктуры разработки и тестирования.
- Должны быть настроены системы мониторинга для сбора метрик производительности и доступности.

См. _«[Развёртывание ПО. Архитектура и ИТ-ландшафт][architecture_landscape]»_ для получения информации о методике расчёта требуемых аппаратных ресурсов и конфигурации системы хранения данных.

### Продолжительность ОПЭ

Рекомендуемая продолжительность опытно-промышленной эксплуатации составляет не менее 1–3 месяцев, в зависимости от сложности приложения и требований заказчика.

### Объем работ на этапе ОПЭ

На этапе ОПЭ должны быть выполнены следующие работы:

- Развертывание приложения в конфигурации ОПЭ.
- Настройка мониторинга и сбора метрик.
- Эксплуатация приложения ограниченным кругом пользователей.
- Сбор данных о производительности, стабильности и использовании ресурсов.
- Документирование выявленных проблем и особенностей эксплуатации.
- Предварительная оценка требований к отказоустойчивости и доступности системы.

!!! note "Примечание"

    На этапе ОПЭ рекомендуется определить уровень критичности проектируемой системы и соответствующие требования к доступности и отказоустойчивости. См. _«[Обеспечение высокой доступности и отказоустойчивости][availability_fault_tolerance]»_ для получения информации о классификации информационных систем по уровням критичности.

### Критерии готовности к переходу на следующий этап

- Приложение стабильно работает в течение установленного периода ОПЭ.
- Собраны данные о производительности и использовании ресурсов.
- Выявленные проблемы устранены или задокументированы.
- Определены предварительные требования к конфигурации для промышленной эксплуатации.

## Этап 4: Нагрузочное тестирование {: #commissioning_recommendations_load_testing }

После завершения ОПЭ клиент самостоятельно должен провести нагрузочное тестирование своих приложений на основе платформы **{{ productName }}**, чтобы определить достаточную, работоспособную, надёжную и устойчивую конфигурацию.

### Требования к нагрузочному тестированию

Нагрузочное тестирование должно включать:

- **Тестирование производительности** — определение максимальной нагрузки, которую может выдержать система при заданной конфигурации.
- **Тестирование стабильности** — проверка стабильной работы системы под нагрузкой в течение продолжительного времени.
- **Тестирование отказоустойчивости** — проверка поведения системы при отказе отдельных компонентов.
- **Тестирование масштабируемости** — проверка возможности увеличения производительности при увеличении ресурсов.

### Сценарии нагрузочного тестирования

Сценарии нагрузочного тестирования должны моделировать реальные условия эксплуатации:

- Типичная нагрузка — нагрузка, соответствующая обычным условиям эксплуатации.
- Пиковая нагрузка — нагрузка, соответствующая пиковым периодам эксплуатации.
- Стресс-нагрузка — нагрузка, превышающая ожидаемую максимальную нагрузку.

### Метрики для оценки результатов

При проведении нагрузочного тестирования должны быть измерены следующие метрики:

- Время отклика системы на запросы пользователей.
- Пропускная способность системы (количество обработанных запросов в единицу времени).
- Использование ресурсов (CPU, память, диск, сеть).
- Количество ошибок и их типы.
- Время восстановления после сбоев.

### Определение конфигурации для промышленной эксплуатации

На основе результатов нагрузочного тестирования определяется конфигурация ПО и оборудования для промышленной эксплуатации. Конфигурация должна обеспечивать:

- Достаточную производительность для обработки ожидаемой нагрузки с запасом не менее 20%.
- Работоспособность системы при типичной и пиковой нагрузке.
- Надёжность системы с учетом требований к отказоустойчивости.
- Устойчивость системы к кратковременным пиковым нагрузкам.

При определении конфигурации следует учитывать:

- Уровень критичности системы и соответствующие требования к доступности (RTO/RPO).
- Требования к отказоустойчивости и высокой доступности.
- Необходимость оптимизации конфигурации вспомогательного ПО.

См. _«[Обеспечение высокой доступности и отказоустойчивости][availability_fault_tolerance]»_ для получения информации о проектировании архитектуры системы и выборе подходов к обеспечению доступности и отказоустойчивости.

См. _«[Развёртывание ПО. Архитектура и ИТ-ландшафт][architecture_landscape]»_ для получения информации о методике расчёта требуемых аппаратных ресурсов.

### Критерии готовности к переходу на следующий этап

- Нагрузочное тестирование проведено в полном объеме.
- Результаты нагрузочного тестирования задокументированы.
- Определена конфигурация для промышленной эксплуатации.

## Этап 5: Промышленная эксплуатация {: #commissioning_recommendations_production }

Для промышленной эксплуатации клиент должен использовать конфигурацию ПО и оборудования, соответствующую результатам нагрузочного тестирования.

### Требования к инфраструктуре

- Конфигурация ПО и оборудования должна соответствовать результатам нагрузочного тестирования.
- Инфраструктура промышленной эксплуатации должна быть изолирована от инфраструктуры разработки, тестирования и ОПЭ.
- Должны быть настроены системы мониторинга, резервного копирования и восстановления.
- Должна быть обеспечена высокая доступность и отказоустойчивость системы в соответствии с уровнем критичности системы.

См. _«[Обеспечение высокой доступности и отказоустойчивости][availability_fault_tolerance]»_ для получения информации о способах повышения доступности и отказоустойчивости, а также о проектировании архитектуры системы.

См. _«[Развёртывание ПО. Архитектура и ИТ-ландшафт][architecture_landscape]»_ для получения информации о рекомендуемых конфигурациях для продуктового ландшафта.

### Оптимизация конфигурации вспомогательного ПО

Для повышения производительности и эффективного использования ресурсов рекомендуется выполнить оптимизацию конфигурации вспомогательного ПО:

- Настройка размера выделяемой области памяти и размера кучи JVM для {{ apacheIgniteVariants }}.
- Настройка количества шардов и размера кучи JVM для {{ openSearchVariants }}.
- Настройка сжатия и количества подключений для {{ nginxVariants }}.

См. _«[Оптимизация конфигурации вспомогательного ПО][auxiliary_software_optimize]»_ для получения подробной информации о настройке параметров вспомогательного ПО.

### Требования к мониторингу

На этапе промышленной эксплуатации должен быть организован непрерывный мониторинг:

- Доступности системы и приложений.
- Производительности системы.
- Использования ресурсов.
- Ошибок и сбоев в работе системы.

### Требования к резервному копированию

Должны быть настроены регулярные резервные копии:

- Базы данных.
- Конфигурации системы.
- Загруженных пользователями файлов.

См. _«[Резервное копирование. Настройка и запуск, просмотр журнала сеансов][backup_configure]»_.

## Этап 6: Ежегодный пересмотр конфигурации {: #commissioning_recommendations_annual_review }

**{{ companyName }}** оставляет за собой право требовать пересмотра конфигураций после выпуска новых версий программного обеспечения, как минимум раз в год.

Раз в год минимальные требования к системным ресурсам могут изменяться с выходом новых сборок платформы.

!!! warning "Внимание"

    Даже если приложения клиента успешно работали на некоторой конфигурации, **{{ companyName }}** может потребовать пересмотра конфигурации после выпуска новых версий программного обеспечения. Минимальная частота пересмотра конфигурации — один раз в год.

### Основания для пересмотра конфигурации

Пересмотр конфигурации может потребоваться в следующих случаях:

- Выпуск новой версии платформы **{{ productName }}** с изменением минимальных требований к системным ресурсам.
- Изменение требований к производительности или доступности системы.
- Изменение объема данных или количества пользователей.
- Выявление проблем с производительностью или стабильностью системы.

### Процедура пересмотра конфигурации

При пересмотре конфигурации должны быть выполнены следующие действия:

1. Анализ текущей конфигурации и её соответствия новым требованиям.
2. Оценка необходимости изменения конфигурации.
3. При необходимости — проведение дополнительного нагрузочного тестирования.
4. Определение новой конфигурации (при необходимости).
5. Планирование и выполнение изменений конфигурации.
6. При необходимости — оптимизация конфигурации вспомогательного ПО.

См. _«[Оптимизация конфигурации вспомогательного ПО][auxiliary_software_optimize]»_ для получения информации о настройке параметров вспомогательного ПО при изменении конфигурации системы.
