# Глоссарий терминов Comindware Business Application Platform {: #glossary }

Здесь представлен глоссарий основных терминов, используемых в **{{ productName }}**.

Для терминов, встречающихся в API, формулах, выражениях N3 и C#-скриптах, по необходимости указаны английские синонимы.

## А

### Аккаунт {: #glossary_account }

Учётная запись пользователя в системе, которая позволяет идентифицировать его и управлять доступом к ресурсам **{{ productName }}**.

Аккаунт содержит данные пользователя, такие как логин, пароль, ФИО, адрес эл.&nbsp;почты, телефон, изображение профиля и т.&nbsp;п.

Аккаунты могут быть привязаны к шаблонам аккаунтов для хранения дополнительных атрибутов.

См.: _[Аккаунты. Администрирование, назначение лицензий][accounts]_

### Атрибут {: #glossary_attribute }

Простейший элемент данных об объекте, аналог поля в реляционной базе данных. Значения атрибутов вводят конечные пользователи, либо они вычисляются автоматически в соответствии с логикой работы приложения. Для ввода и просмотра значений атрибутов используются поля на формах и в таблицах. Атрибуты предусмотрены у шаблона записи, шаблона аккаунта, шаблона роли и шаблона организационной единицы.

См.: _[Атрибуты. Определения, типы, настройка, архивирование, удаление][attributes]_

Термины в API:

- **property**.

## В

### Виджет {: #glossary_widget }

Виджет — это визуальный элемент на информационной странице, который отображает данные или показатели в удобном для восприятия виде: диаграммы, списки, карточки, временные шкалы, показатели и т.&nbsp;п. Набор и расположение виджетов определяется настройкой страницы.

См.: _[Информационные страницы. Определения и настройка][pages_setup]_

### Витрина {: #glossary_showcase }

Витрина — это тип информационного блока на странице, где данные отображаются в виде набора виджетов, карточек или плиток для наглядного анализа. На витрине можно настроить фильтры, представления и состав виджетов для различных сценариев работы пользователей.

См.: _[Информационные страницы. Определения и настройка][pages_setup]_

### Вкладка {: #glossary_tab }

Вкладка — это область на форме или странице, которая позволяет сгруппировать поля, вложенные формы и другие элементы по смысловым блокам. Вкладки помогают упростить восприятие формы и переключаться между наборами полей без перехода на другие страницы.

См.: _[Статические элементы формы: области, вкладки, колонки, статичный текст. Настройка представления][form_static_elements]_

### Выражение {: #glossary_expression }

Выражения используются для вычисления и проверки значений атрибутов и переменных, в правилах для форм, сценариях и скриптах, а также для выборки и обработки данных. В платформе предусмотрен редактор выражений в следующих нотациях: язык формул, язык N3, язык C# и таблицы DMN.

См.: _[Редактор выражений. Использование][expression_editor]_

## Д

### Диаграмма процесса {: #glossary_process_diagram }

Диаграмма процесса служит для моделирования и исполнения бизнес-процесса. Она привязана к шаблону процесса, создаётся при создании шаблона процесса, строится в нотации BPMN 2.0 и задаёт последовательность выполнения задач и действий в рамках бизнес-процесса. Диаграмма процесса позволяет моделировать и исполнять бизнес-процесс с назначением задач исполнителям, выполнением сценариев, отправкой внутренних и внешних сообщений, просмотром журнала действий.

См.: _[Диаграмма процесса. Определения, просмотр, редактирование, публикация][process_diagram]_

Термины в API:

- **scheme**.

## З

### Запись {: #glossary_record }

Запись — это набор данных (атрибутов) об одной бизнес-сущности и представляет собой аналог строки в реляционной базе данных. Записи могут быть представлены в виде таблиц и форм.

См.: _[Шаблон записи][record_templates]_

Термины в API:

- **record**.

### Задача {: #glossary_task }

Задача — это элемент бизнес-процесса, который должен быть выполнен пользователем или системой. Пользовательские задачи назначаются исполнителям и отображаются в списке задач пользователя. При завершении задачи токен процесса переходит к следующему элементу диаграммы процесса. Задачи могут содержать формы для ввода данных и правила для их обработки.

См.: _[Пользовательская задача][process_diagram_elements_user_task]_

## И

### Идентификатор {: #glossary_identifier }

Идентификатор — уникальный признак объекта, по которому его можно отличить от других. Аналогичен ключевому полю в реляционных базах данных. Идентификатор представляет собой строку с номером объекта и формируется автоматически для всех объектов при их создании. При экспорте и импорте приложений идентификаторы назначаются заново, поэтому выражения, в которых они использовались, потребуется скорректировать соответствующим образом.

См.: _[Идентификаторы и системные имена][identifiers_system_names]_

## К

### Кнопка {: #glossary_button }

Кнопка представляет собой визуальный элемент на форме, при нажатии которого выполняется предварительно настроенная операция. Для каждого шаблона предусмотрен отдельный набор кнопок. Названия, цвета и значки кнопок настраиваются в области кнопок.

См.: _[Кнопки. Определение, настройка, удаление][buttons]_

Термины в API:

- **user command**.

## П

### Переменная приложения {: #glossary_variable }

Переменные приложения — это атрибуты с заданным значением, которые не привязаны к какому-либо шаблону. Их можно использовать в C#-скриптах и выражениях на языке N3 в любом шаблоне или сценарии. Переменные одного приложения доступны во всех остальных приложениях.

См.: _[Переменные приложения. Просмотр списка, настройка и использование][variables]_

### Пользователь {: #glossary_user }

Пользователь — это человек, работающий в системе под своим аккаунтом и выполняющий задачи в приложениях. Доступ пользователя к данным и функциям системы определяется его аккаунтом, назначенными ролями и правами доступа.

См.: _[Аккаунты. Администрирование, назначение лицензий][accounts]_

### Права доступа {: #glossary_access_rights }

Права доступа — это совокупность всех разрешений, предоставленных аккаунту или группе аккаунтов. Права доступа формируются посредством ролей, правил для форм, выражений, языка запросов и скриптов.

См.: _[Роли в приложении. Определения, настройка, объединение, удаление][roles]_

### Поле {: #glossary_field }

Поле — это элемент формы или таблицы, привязанный к атрибуту и предназначенный для ввода и отображения значений атрибута. При перетаскивании атрибута на форму создаётся поле, представление которого зависит от типа атрибута. Поля могут иметь различные режимы доступа: требовать ввод, разрешать ввод, только чтение или быть скрытыми.

См.: _[Динамические элементы формы: поля атрибутов и вложенные формы. Настройка представления][form_dynamic_elements]_

### Приложение {: #glossary_application }

Приложение — это настроенная в **{{ productName }}** совокупность бизнес-процессов, диаграмм, моделей данных, экранных форм, механизмов обработки данных и ролевой модели для решения определённых задач. Как правило, в рамках одного приложения реализуют набор процессов, связанных между собой общей бизнес-задачей.

См.: _[Приложения. Определения, создание, настройка, администрирование, удаление][apps]_

Термины в API:

- **solution**;
- **container**.

### Процесс {: #glossary_process }

Процесс — это экземпляр бизнес-процесса, запущенный на основе шаблона процесса. Процесс выполняется по диаграмме процесса и может быть запущен по записям, из списка записей, из других процессов или вручную. В ходе выполнения процесса данные хранятся в связанном шаблоне записи, выполняются задачи, сценарии и скрипты. Процессы могут взаимодействовать друг с другом и с внешними системами.

См.: _[Шаблон процесса][process_templates]_

## Р

### Раздел навигации {: #glossary_navigation_section }

Раздел навигации — это набор пунктов, позволяющих переходить к различным страницам. Разделы навигации отображаются как пункты в панели навигации. В состав раздела навигации могут входить подразделы, ссылки на страницы, страница «Мои задачи», рабочий стол и гиперссылки на внутренние и внешние веб-страницы.

См.: _[Панель и разделы навигации. Определения, настройка, удаление][navigation_sections_setup]_

### Роль {: #glossary_role }

Роль — это совокупность разрешений, назначенная аккаунту или группе аккаунтов. Разрешение — это полномочие на совершение определенного действия с объектом: полный доступ, создание, просмотр, изменение, удаление и т.&nbsp;п.

См.: _[Роли в приложении. Определения, настройка, объединение, удаление][roles]_

### Разрешение {: #glossary_permission }

Разрешение — это полномочие на совершение определённого действия с объектом: полный доступ, создание, просмотр, изменение, удаление и т.&nbsp;п. Разрешения объединяются в роли и таким образом определяют, какие действия может выполнять пользователь в приложении.

См.: _[Роли в приложении. Определения, настройка, объединение, удаление][roles]_

## С

### Системное имя {: #glossary_system_name }

Системное имя — признак объекта, уникальный и отличающий его от других объектов в заданном контексте. Системное имя аналогично имени объекта или переменной в программировании. Системное имя задаёт создатель объекта и является более наглядным, чем идентификатор. При экспорте и импорте приложений системные имена остаются неизменными, и корректировать выражения с ними не требуется.

См.: _[Идентификаторы и системные имена][identifiers_system_names]_

Термины в API:

- **alias**;
- **global alias** (глобальный псевдоним) — путь к объекту, состоящий из его типа, имени родительской системы и имени системы объекта.

### Связанная запись {: #glossary_related_record }

Связанная запись — это запись из другого шаблона записи, на которую ссылается атрибут типа «Запись». Атрибут типа «Запись» содержит одну или несколько ссылок на записи в связанном шаблоне записи. Связанные записи могут отображаться в полях на формах и в таблицах в виде гиперссылок или простого текста с заголовком записи.

См.: _[Атрибут типа «Запись»][attribute_record]_

### Сценарий {: #glossary_scenario }

Сценарий представляет собой алгоритм автоматизации операций, проверки, обработки и передачи данных, который запускается при наступлении заданного события и выполняет настроенную последовательность действий. Помимо обработки данных внутри **{{ productName }}** сценарии позволяют обмениваться данными с внешними системами. Сценарий составляется из блоков с помощью визуального конструктора.

См.: _[Сценарии. Определения, создание, настройка, использование][scenarios]_

Термины в API:

- **event**;
- **trigger**;
- **behaviour**.

## Т

### Таблица {: #glossary_table }

Таблица — это представление данных в виде столбцов и строк. Строка таблицы содержит данные одной записи шаблона. Столбец таблицы содержит значения атрибутов записи. В каждом шаблоне можно настроить произвольное количество таблиц. В API и некоторых контекстах таблица может называться View.

См.: _[Таблицы. Определения, настройка, клонирование, удаление][table_configure]_

Термины в API:

- **dataset**;
- **vew**.

### Столбец {: #glossary_column }

Столбец таблицы содержит значения одного атрибута для всех записей, отображаемых в таблице. Пользователь может настраивать столбцы: менять их порядок, скрывать, группировать строки и подсчитывать итоги по значениям в столбце.

См.: _[Таблицы. Определения, настройка, клонирование, удаление][table_configure]_

### Токен {: #glossary_token }

Токен — это маркер текущей позиции выполнения процесса в нотации BPMN. На диаграмме процесса может быть один или несколько токенов, каждый из которых движется по своему пути. Токены создаются на начальных событиях, некоторых промежуточных событиях и расходящихся развилках, а удаляются на конечных событиях и сходящихся развилках.

См.: _[Диаграмма процесса. Определения, просмотр, редактирование, публикация][process_diagram]_

## Ф

### Форма {: #glossary_form }

Форма представляет собой веб-страницу для ввода и отображения данных. На форме можно разместить поля, привязанные к атрибутам, вложенные формы, статичный текст, а также различные визуальные элементы.

См.: _[Формы. Определения, редактирование, удаление][forms]_

### Флажок {: #glossary_checkbox }

Флажок — это элемент управления на форме или в таблице, который показывает, включено ли определённое свойство или параметр. Пользователь может установить или снять флажок, чтобы включить или отключить соответствующую опцию. В интерфейсе рекомендуется использовать термин «флажок», а не разговорное «галочка».

См.: _[Атрибут типа «Логический»][attribute_boolean]_

## Ш

### Шаблон {: #glossary_template }

Шаблоны служат для описания, хранения и обработки данных бизнес-сущностей и процессов. Для каждого приложения предусмотрен отдельный набор шаблонов. В платформе предусмотрены шаблоны следующих типов: шаблон записи, шаблон процесса, шаблон аккаунта, шаблон роли и шаблон организационной единицы.

См.: _[Шаблоны. Определения, настройка, перенос, архивирование, очистка][templates]_

Термины в API:

- **template**;
- **container**.

### Шаблон аккаунта {: #glossary_account_template }

Шаблон аккаунта — это специальный шаблон записи, в котором хранятся особые атрибуты аккаунтов. Используйте шаблоны аккаунтов для настройки и хранения произвольных атрибутов, форм и таблиц аккаунтов.

См.: _[Шаблон аккаунта][account_templates]_

### Шаблон записи {: #glossary_record_template }

Шаблоны записей используются для построения модели, хранения и обработки данных, описывающих бизнес-сущности. Шаблон записи служит для настройки свойств шаблона, атрибутов и записей, а также для хранения записей. Он представляет собой аналог таблицы в реляционной базе данных.

См.: _[Шаблон записи][record_templates]_

Недопустимые термины:

- Record type — недопустимое название шаблона записи.

### Шаблон процесса {: #glossary_process_template }

Шаблоны процессов используются для описания и исполнения бизнес-процессов. Шаблон процесса состоит из диаграммы процесса в нотации BPMN 2.0, данных и других свойств, определяющих, как должны выполняться все процессы определенного вида. Каждый шаблон процесса должен и может быть связан только с одним шаблоном записи.

См.: _[Шаблон процесса][process_templates]_

## Э

### Экземпляр процесса {: #glossary_process_instance }

Экземпляр процесса — это конкретный запущенный процесс, созданный на основе шаблона процесса. Каждый экземпляр процесса выполняется по определённой версии диаграммы процесса и имеет собственные данные, хранящиеся в связанном шаблоне записи. На диаграмме экземпляра процесса можно просмотреть текущие позиции токенов, создать, перезапустить или удалить токены, остановить процесс и просмотреть цепочку событий.

См.: _[Использование диаграммы экземпляра процесса][process_diagram_view_instance]_

## Идентификаторы объектов {: #glossary_ids }

Идентификатор — уникальный признак объекта, по которому его можно отличить от других. Он формируется автоматически при создании объекта и представляет собой строку с номером объекта (`<XX>` — номер объекта). Идентификаторы используются в выражениях N3 и формул, C#-скриптах, вызовах API, HTTP- и SQL-запросах.

В **{{ productName }}** используются следующие форматы идентификаторов:

- Аккаунт: `account.<XX>`
- Атрибут: `op.<XX>`
- Группа: `group.<XX>`
- Запись, процесс, задача: `<XX>`
- Карточка: `card.<XX>`
- Кнопка: `event.<XX>`
- Приложение: `sln.<XX>`
- Роль: `role.<XX>`
- Сценарий: `trigger.<XX>`
- Таблица: `lst.<XX>`
- Форма: `form.<XX>`
- Шаблон записи: `oa.<XX>`
- Шаблон процесса: `pa.<XX>`

Подробнее см.: [Идентификаторы и системные имена][identifiers_system_names]

## Термины, используемые в API {: #glossary_api_only }

В документации API и Swagger используются следующие термины:

- **activity** — элемент диаграммы процесса;
- **alias** — системное имя;
- **behaviour** — сценарий, элемент диаграммы процесса;
- **container** — шаблон, приложение;
- **dataset** — таблица в шаблоне;
- **global alias** (глобальный псевдоним) — путь к объекту, состоящий из его типа, имени родительской системы и имени системы объекта;
- **inline** — особый тип документа, который отображается на форме как текстовый блок и который можно редактировать в форме;
- **list** — массив;
- **object** — набор пар свойство/значение;
- **property** — атрибут;
- **record** — запись;
- **revision** — версия файла, прикреплённого к атрибуту;
- **scheme** — диаграмма процесса;
- **solution** — приложение;
- **stream** — файл, хранящийся в системе;
- **template** — шаблон;
- **transfer** — управление версиями приложения;
- **trigger** — термин в продукте не используется, но есть в URL и некоторых API;
- **user command** — кнопка;
- **view** — таблица;
- **worker** — системный сервис.
