# Методология управления версиями приложений {: #version_control_methodology }

## Введение {: #version_control_methodology_intro }

В этой статье описана рекомендуемая методология настройки и сопровождения приложений в **{{ productName }}** с использованием управления версиями.

Методология дополняет функциональные инструкции из статей [«Управление версиями приложения. Экспорт и импорт»][version_control], [«Экспорт и импорт версий приложения с помощью {{ gitVariants }}»][version_control_git], [«Экспорт и импорт версий приложения вручную»][version_control_manual] и [«Экспорт и импорт версий приложения с помощью Excel»][version_control_excel] и объясняет общий процесс работы с версиями, средами и релизами.

Рекомендации из этой статьи применимы независимо от выбранного способа экспорта и импорта версий приложения (через {{ gitVariants }}, файл `CTF` или файл Excel).

## Среды и ветки {: #version_control_methodology_envs_branches }

### Рекомендуемые среды

Для каждого значимого приложения рекомендуется использовать как минимум две среды:

- **среда разработки** — экземпляр **{{ productName }}**, в котором выполняются настройка, отладка и проверка изменений;
- **продуктивная среда** — экземпляр **{{ productName }}**, в котором приложение используется конечными пользователями.

При необходимости могут использоваться дополнительные среды (например, тестовая или предпродуктивная), однако базовый принцип остаётся неизменным: изменения сначала выполняются и проверяются в отдельной среде, а затем переносятся в продуктив.

### Рекомендуемая схема веток

При использовании {{ gitVariants }} рекомендуется настраивать репозиторий с как минимум двумя ветками:

- **рабочая ветка разработки** (например, `develop`) — для текущих изменений и интеграции задач;
- **стабильная ветка продуктивных версий** (например, `master`) — для версий, которые развернуты или планируются к развертыванию в продуктивной среде.

В простой схеме экземпляр среды разработки связан с рабочей веткой, а продуктивный экземпляр — со стабильной веткой. При этом:

- изменения фиксируются в рабочей ветке;
- перед выпуском новой версии приложения изменения из рабочей ветки сливаются в стабильную ветку;
- импорт в продуктив выполняется из стабильной ветки.

### Запрет прямых изменений в продуктивной среде

Рекомендуется придерживаться следующих правил:

- настройки приложения выполняются только в среде разработки;
- любые изменения конфигурации на продуктивной среде допускаются только в исключительных случаях и должны быть затем воспроизведены в среде разработки и перенесены через механизм управления версиями;
- стабильная ветка репозитория (например, `master`) не используется для непосредственной разработки — изменения попадают в неё только через слияние из рабочей ветки.

## Цикл настройки приложения {: #version_control_methodology_cycle }

### Допустимые типы изменений

В рамках одного цикла настройки могут изменяться различные элементы конфигурации приложения, в том числе:

- бизнес-процессы и диаграммы процессов;
- формы и правила для форм;
- таблицы и фильтры;
- атрибуты и вычисления;
- шаблоны записей, включая справочники;
- сценарии и параметры приложения;
- разделы навигации и страницы.

### Логически завершённые изменения

Рекомендуется группировать изменения в логически завершённые блоки. Для каждого блока:

- формулируется цель (например, реализация конкретной задачи или пользовательского сценария);
- изменения не смешиваются с другими, не связанными задачами;
- результат можно проверить отдельно от других работ.

Это упрощает анализ коммитов, поиск причин ошибок и откаты к предыдущим версиям.

### Проверка изменений в среде разработки

После завершения логического блока изменений выполните проверку работоспособности решения в среде разработки. В проверку, как правило, входят:

- корректность выполнения процессов;
- отображение форм и таблиц;
- работа правил;
- выполнение сценариев и интеграций;
- воспроизведение ключевых пользовательских сценариев.

При выявлении ошибок изменения дорабатываются и цикл проверки повторяется.

### Фиксация изменений (коммиты) {: #version_control_methodology_commits }

После того как изменения успешно проверены в среде разработки, фиксируйте их как новую версию.

При использовании {{ gitVariants }}:

- выполняется экспорт приложения в репозиторий из среды разработки;
- в выбранной ветке создаётся коммит с конфигурацией приложения.

Сообщения коммитов рекомендуется формулировать так, чтобы по ним можно было понять суть изменений без изучения конфигурации. Например:

- использовать ссылки на задачи из системы управления задачами;
- кратко описывать изменённый функционал или область приложения.

При ручном экспорте в `CTF` или Excel:

- сохраняйте связь между файлами экспортированных версий и соответствующими задачами и релизами;
- используйте понятную схему именования файлов и каталогов (например, с указанием номера версии и цели изменений).

## Подготовка и перенос изменений в продуктив {: #version_control_methodology_transfer_to_prod }

### Подготовка к релизу

Перед переносом изменений в продуктивную среду рекомендуется:

- убедиться, что рабочая ветка с изменениями актуальна и включает все необходимые коммиты;
- согласовать с командой параллельные работы, чтобы избежать конфликтов и неожиданных изменений;
- зафиксировать состав изменений, входящих в релиз (например, список задач или требований).

### Слияние веток и формирование релиза

При использовании {{ gitVariants }} типичный порядок действий:

1. Подготовить изменения в рабочей ветке (например, `develop`) и убедиться, что они успешно проходят проверку в среде разработки.
2. Создать запрос на слияние (pull request) из рабочей ветки в стабильную ветку (например, `master`).
3. Проанализировать список изменений, при необходимости устранить конфликты.
4. Выполнить слияние в стабильную ветку.
5. Установить тег версии в стабильной ветке (см. раздел «Версионирование и откат»).

### Обновление продуктивной среды

После формирования новой версии:

- перенесите версию приложения в продуктивную среду с помощью выбранного механизма управления версиями ({{ gitVariants }}, файл `CTF` или файл Excel);
- используйте те же принципы подготовки, что описаны в разделе [«Подготовка приложения к экспорту»][version_control_app_prepare];
- выполните контрольную проверку по заранее подготовленным тест-кейсам или чек-листам.

## Версионирование и откат {: #version_control_methodology_versions }

### Фиксация версий приложения

Каждое обновление продуктивной среды рекомендуется рассматривать как отдельную версию приложения.

При использовании {{ gitVariants }} это обычно реализуется с помощью тегов в стабильной ветке. Например:

- тег содержит номер версии приложения и при необходимости краткое описание;
- тег ставится на коммит, соответствующий версии, развернутой в продуктивной среде.

Кроме того, номер версии следует фиксировать в сопровождающей документации решения и, при необходимости, в самом приложении.

### Откат к предыдущей версии

При необходимости возврата к предыдущей версии рекомендуется:

- определить целевую версию (тег или коммит, соответствующий стабильному состоянию приложения);
- подготовить конфигурацию этой версии (через {{ gitVariants }}, файл `CTF` или файл Excel);
- импортировать выбранную версию в продуктивную среду;
- выполнить проверку работоспособности приложения после отката.

При использовании {{ gitVariants }} откат версии конфигурации в репозитории выполняется на стороне системы контроля версий, а в **{{ productName }}** импортируется соответствующая версия приложения.

## Особенности импорта и экспорта и ограничения {: #version_control_methodology_transfer_behavior }

Общие особенности поведения при экспорте и импорте, а также перечень сущностей, которые переносятся или не переносятся, подробно описаны в статье [«Управление версиями приложения. Экспорт и импорт»][version_control].

При планировании релизов и анализе последствий импорта версии приложения учитывайте, что, в частности:

- шаблоны, отсутствующие в импортируемой версии, но присутствующие в текущем приложении, могут быть признаны неактуальными и удалены вместе с записями;
- некоторые сущности не экспортируются и должны настраиваться отдельно (например, аккаунты, группы, системные роли, подключения, отдельные пути передачи данных и элементы навигации);
- при импорте переносятся опубликованные диаграммы процессов, при этом старые экземпляры процессов продолжают выполняться по прежним версиям диаграмм;
- обсуждения и часть истории могут не переноситься.

Перед каждым импортом версии приложения рекомендуется предварительно оценить влияние операции на связанные приложения и внешние интеграции.

Подробную информацию о влиянии экспорта и импорта на идентификаторы и системные имена см. в статье [«Идентификаторы и системные имена»][identifiers_system_names].

## Типовые ошибки и рекомендации {: #version_control_methodology_common_errors }

В процессе управления версиями приложений часто встречаются следующие ситуации:

- попытка экспорта или импорта при некорректных настройках подключения к репозиторию или ограничениях доступа;
- отсутствие фактических изменений в конфигурации приложения при попытке выполнить экспорт;
- использование идентификаторов объектов в выражениях, которые становятся недействительными после импорта в другой экземпляр;
- экспорт приложений со связанными шаблонами из других приложений без предварительной подготовки;
- смешивание нескольких несвязанных задач в одном релизе или коммите, что усложняет анализ и откат.

Чтобы снизить риски:

- заранее готовьте приложения к экспорту согласно разделу [«Подготовка приложения к экспорту»][version_control_app_prepare];
- используйте системные имена вместо идентификаторов в выражениях, когда это возможно;
- разбивайте крупные изменения на несколько версий с промежуточными проверками;
- избегайте выполнения настройки непосредственно на продуктивной среде;
- фиксируйте информацию о составе изменений каждой версии.

## Подключение управления версиями к уже работающему продуктиву {: #version_control_methodology_existing_prod }

Если приложение уже используется в продуктивной среде, но управление версиями еще не было настроено, рекомендуется следующий подход:

1. Создайте среду разработки, максимально соответствующую текущему состоянию продуктивной среды. Для этого:
   - выполните резервное копирование продуктивного экземпляра **{{ productName }}**;
   - разверните копию на отдельном сервере или в отдельном экземпляре, руководствуясь инструкциями по резервному копированию и восстановлению.
2. В среде разработки устраните ошибки и несоответствия, которые могут препятствовать экспорту приложения.
3. Настройте подключение к репозиторию {{ gitVariants }}, если планируется использование Git. См. статью [«Управление версиями через Git. Настройка подключения»][git_connection].
4. Включите управление версиями для приложения и выполните первый экспорт в репозиторий или в файл версии.
5. После успешного экспорта начните использовать описанную в этой статье методологию для дальнейших изменений и релизов.

Такой подход позволяет максимально безопасно перевести уже используемое приложение на управление версиями без прямых экспериментов в продуктивной среде.
