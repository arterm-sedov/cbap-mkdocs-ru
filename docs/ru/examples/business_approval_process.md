# Универсальный процесс согласования. Настройка шаблонов, процессов и сценария {: #business_approval_process }

Во многих процессах есть одинаковый блок согласования: например, заявку на закупку нужно сперва согласовать, а затем направить на оценку и закупку; договор перед подписанием также проходит согласование.

Ниже приведён пример настройки универсального процесса согласования, который можно переиспользовать для разных объектов: заявок, договоров и других мастер-данных.

## Настройка шаблона истории согласований {: #business_approval_process_history_template }

1. Создайте шаблон записи _«История согласований»_.
2. Создайте в нём следующие атрибуты:

| Атрибут | Настройка |
| --- | --- |
| _Решение_ | Атрибут типа «**Запись**», связанный с шаблоном, где хранятся варианты решений для согласования. |
| _Комментарий к решению_ | Атрибут типа «**Текст**». |
| _Ответ на запрос_ | Атрибут типа «**Текст**». |
| _Дата и время начала_ | Атрибут типа «**Дата и время**». |
| _Дата и время завершения_ | Атрибут типа «**Дата и время**». |
| _ID объекта согласования_ | Атрибут типа «**Текст**». |
| _Название объекта согласования_ | Атрибут типа «**Текст**». |
| _Объект согласования_ | Атрибут типа «**Текст**» с форматом отображения «**HTML-текст**», вычисляемый по формуле, приведённой ниже. |
| _Инициатор_ | Атрибут типа «**Аккаунт**». |
| _Согласующий_ | Атрибут типа «**Аккаунт**». |
| _Статус_ | Атрибут типа «**Запись**», связанный с шаблоном, где хранятся статусы согласования. |
| _Повторное согласование_ | Атрибут типа «**Запись**», связанный с шаблоном _История согласований_, с взаимной связью по новому атрибуту _Изначальное согласование_. |
| _История согласования_ | Атрибут типа «**Запись**», связанный с шаблоном _История согласований_, с несколькими значениями и вычислением по формуле, приведённой ниже. |
| _Атрибуты для мастер-данных_ | Атрибуты типа «**Запись**», связанные с шаблонами, где хранятся ваши мастер-данные (например, заявки или договоры), с взаимной связью по новому атрибуту _История согласования_ с несколькими значениями. |

3. Для атрибута _«Объект согласования»_ задайте следующую формулу:

```sql
FORMAT("<a href='/#Resolver/{0}' target='_blank'>{1}</a>", LIST($idobektasoglasovaniya,$Nazvanieobektasoglasovaniya))
```

4. Для атрибута _«История согласования»_ задайте следующую формулу:

```sql
from a in db->Istoriyasoglasovanij where (a->idobektsoglasovaniya == $idobektsoglasovaniya) select a->id
```

5. В шаблоне _«История согласований»_ настройте формы:

    - форму для отображения по умолчанию;
    - форму для задачи согласующему;
    - форму для задачи инициатору.

## Настройка процесса согласования {: #business_approval_process_approval_process }

1. Настройте диаграмму процесса по образцу:

    _![Пример диаграммы процесса согласования](https://kb.comindware.ru/assets/2023-02-13_14h44_52.png)_

2. Добавьте сценарий на входе у первой задачи.
3. В сценарии на входе добавьте действие «**Изменить значения атрибутов**» и настройте его:

| Атрибут | Значение |
| --- | --- |
| _Статус_ | Используйте функцию `OBJECT`, чтобы найти необходимый статус. |
| _Дата и время начала_ | Используйте функцию `NOW`. |

4. Добавьте сценарий на выходе у первой задачи.
5. В сценарии на выходе добавьте действие «**Изменить значения атрибутов**» и настройте его:

| Атрибут | Значение |
| --- | --- |
| _Статус_ | Используйте функцию `OBJECT`, чтобы найти необходимый статус. |
| _Дата и время завершения_ | Используйте функцию `NOW`. |

6. Аналогично настройте изменение статуса строки истории согласования на других элементах процесса.

## Настройка повторного согласования {: #business_approval_process_repeated_approval }

1. Добавьте сценарий на входе в элемент вызова процесса и настройте его:

    _![Сценарий на входе](https://kb.comindware.ru/assets/2023-02-20_10h32_27.png)_

2. Добавьте действие «**Создать запись**» и настройте его:

| Поле | Значение |
| --- | --- |
| **Целевой шаблон записи** | Шаблон записи _История согласований_. |
| **Ссылка на новую запись** | Ссылка на целевой шаблон записи в текущем шаблоне: _Повторное согласование_. |
| **Операция со значениями** | «**Добавить**». |

3. Добавьте действие «**Изменить значения атрибутов**».
4. Настройте передачу данных из контекста изначального согласования в контекст новой строки истории согласования:

| Атрибут | Значение |
| --- | --- |
| _Согласующий_ | Значение одноимённого атрибута из изначального согласования. |
| _Инициатор_ | Значение одноимённого атрибута из изначального согласования. |
| _ID объекта согласования_ | Значение одноимённого атрибута из изначального согласования. |
| _Название объекта согласования_ | Значение одноимённого атрибута из изначального согласования. |
| _Заявка_ | Значение одноимённого атрибута из изначального согласования. |

!!! tip "Если согласующих несколько"

    Добавьте действие «**Цикл по объектам**», укажите в нём атрибут или выражение для поиска согласующих и задайте переменную.

    Затем в действии «**Изменить значения атрибутов**» вместо поля _Согласующий_ используйте созданную переменную. Обращение к переменной начинается с `$$`.

5. Настройте вызов процесса повторного согласования:

| Поле | Значение |
| --- | --- |
| **Записи для запуска процесса** | Атрибут _Повторное согласование_. |
| **Шаблон вызываемого процесса** | Процесс согласования, настроенный в разделе [«Настройка процесса согласования»](#business_approval_process_approval_process). |

6. Настройте логику назначения задач.
7. Поместите на задачи соответствующие формы из связанного шаблона записи.
8. Опубликуйте процесс.

## Настройка основного процесса {: #business_approval_process_main_process }

1. Откройте шаблон записи, где хранятся мастер-данные, например заявки на закупку.
2. Поместите атрибут _История согласования_ на все необходимые формы.
3. Настройте отображение таблицы для атрибута _История согласования_.
4. Откройте диаграмму процесса, связанного с заявками на закупку.
5. Добавьте вызов процесса:

    _![Диаграмма основного процесса](https://kb.comindware.ru/assets/2023-02-20_10h08_49.png)_

6. Добавьте сценарий на входе в элемент вызова процесса и настройте его:

    _![Сценарий на входе](https://kb.comindware.ru/assets/2023-02-13_14h51_23.png)_

7. Добавьте действие «**Создать запись**» и настройте его:

| Поле | Значение |
| --- | --- |
| **Целевой шаблон записи** | Шаблон записи _История согласований_. |
| **Ссылка на новую запись** | Ссылка на целевой шаблон записи в текущем шаблоне: _История согласования_. |
| **Операция со значениями** | «**Добавить**». |

8. Добавьте действие «**Изменить значения атрибутов**».
9. Настройте передачу данных из контекста заявки в контекст строки истории согласования:

| Атрибут | Значение |
| --- | --- |
| _Согласующий_ | Значение необходимого атрибута из заявки. |
| _Инициатор_ | Значение необходимого атрибута из заявки. |
| _ID объекта согласования_ | ID заявки, вычисленный с помощью функции `FORMAT`. |
| _Название объекта согласования_ | Название заявки. |

10. Настройте вызов процесса согласования:

| Поле | Значение |
| --- | --- |
| **Записи для запуска процесса** | Атрибут _История согласования_. |
| **Шаблон вызываемого процесса** | Процесс согласования, настроенный в разделе [«Настройка процесса согласования»](#business_approval_process_approval_process). |
| **Выполнение экземпляров процесса** | Параллельное или последовательное выполнение. |
| **Условие запуска процесса** | `EMPTY($Status)`, чтобы согласования запускались только по новым строкам истории согласования. |

11. Настройте логику последующей развилки.

    Например, если в истории согласования нет ни одного отрицательного решения, значит заявка согласована. Можно также настроить отдельный атрибут и передавать в него значение при завершении каждого согласования.

12. Опубликуйте процесс и протестируйте.

Настройки основного процесса можно повторить для всех шаблонов с мастер-данными, где требуется согласование.
