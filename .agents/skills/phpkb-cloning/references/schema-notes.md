# PHPKB Schema Notes

These notes summarize the CMW PHPKB schema surface relevant to cloning.

Generated schema report:

- `phpkb_schema_cmw.md`
- Profile: `cmw`
- Database: `phpkbv9`
- Tables: 39
- Columns: 303
- Declared foreign keys: 0

The database uses MyISAM tables and does not declare foreign keys, so clone scripts must preserve relationships explicitly.

## Currently Cloned Tables

`utilities/phpkb_cloning/phpkb_clone.py` clones:

- `phpkb_categories`
- `phpkb_articles`
- `phpkb_relations`
- `phpkb_attachments`
- `phpkb_custom_data`

## Article Row

`phpkb_articles` is cloned as a full article row except `article_id`, which is regenerated by MySQL auto-increment.

Relevant visibility columns:

- `article_show` (`yes` / `no`) ‚Äî whether the article is shown in KB navigation.
- `unlisted` (`0` / `1`) ‚Äî direct-link-only articles; not used as a clone filter.

The script may adjust:

- `article_title`, when cloning a specific article with a suffix.
- `article_show`, when cloning with `--article-id` without `--show` (defaults to hidden).

Category-tree clones copy `unlisted` from the source row. They only walk articles with
`article_show='yes'`. MkDocs front matter `unlisted: true` is unrelated to cloning; root-level
`phpkb_update_articles.py` pushes PHPKB `unlisted` from export HTML `kb-unlisted="1"`, which
MkDocs emits when front matter has `unlisted: true`.

## Category Row

`phpkb_categories` is cloned as a full category row except `category_id`, which is regenerated by MySQL auto-increment.

Relevant visibility columns:

- `category_status` (`public` / `private`) ‚Äî private subtrees are skipped unless
  `--include-private` is passed to `phpkb_clone.py`.
- `category_show` (`yes` / `no`) ‚Äî hidden categories are never walked.

Child-category traversal also requires `language_id = 2` (Russian). Articles are not
filtered by `language_id`; they follow `phpkb_relations` under the walked category tree.

The script updates:

- `parent_id`, so the cloned category is placed under the cloned or selected parent category.

## Article-Category Relation

`phpkb_relations` is not copied wholesale.

The script inserts a new relation with:

- new `article_id`
- new or selected `category_id`
- copied `article_priority`

## Attachments

`phpkb_attachments` columns:

- `attachment_id`
- `article_id`
- `file_guid`
- `file_name`
- `file_type`
- `file_views`
- `file_status`

The script copies attachment metadata rows and remaps `article_id` to the cloned article.

Physical files are not duplicated. The cloned attachment rows keep the same `file_guid`, so the cloned article links back to the same stored file.

## Custom Data

`phpkb_custom_data` columns:

- `data_id`
- `field_id`
- `article_id`
- `field_data`

The script copies custom data rows and remaps `article_id` to the cloned article.

`field_id` is preserved because it references existing custom field definitions.

## Deliberately Not Cloned

Do not clone these tables unless the business requirement is explicit:

- `phpkb_article_versions`
- `phpkb_article_feedback`
- `phpkb_article_visits`
- `phpkb_comments`
- `phpkb_favorites`
- `phpkb_ratings`
- `phpkb_subscribers`

These tables represent history, behavior, or user activity rather than article-owned content.
